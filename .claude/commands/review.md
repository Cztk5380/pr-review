请对以下 GitCode PR 进行代码检视：$ARGUMENTS

## 执行步骤

1. 在项目根目录执行 PR 审查脚本，根据操作系统选择合适命令：
   - **Windows**：`py review_draft_cursor.py --pr "$ARGUMENTS" --backend agent`
   - **macOS / Linux**：`python3 review_draft_cursor.py --pr "$ARGUMENTS" --backend agent`
   - 若已激活虚拟环境，直接用 `python review_draft_cursor.py ...`

2. 若脚本输出 `[SKIP]`，说明本次 PR diff 与上次审查完全相同，直接告知用户无需重新审查。

3. 读取 `output/` 目录下最新生成的 `review_task_*.md` 文件（按文件名时间戳排序取最新）。

4. 若存在 `line_candidates_*.md`，一并读取用于行级定位。

5. 基于任务文件输出完整的检视意见，要求：
   - **按严重级别排序**（高 → 中 → 低）
   - 每条意见必须严格使用以下模板（便于一键复制）：

```
[文件:<path>] [行号:<start>-<end>] [严重级别:<高/中/低>] 【review】<一句话标题> <具体问题点 + 影响范围 + 触发场景，自然融入描述，禁止显式写「触发条件：」标签；证据不足时写"需人工确认"> 修改建议：<改法描述，代码必须使用行内代码 `改动示例` 格式，禁止只写文字>
```

   - 仅输出有 diff 证据支撑的问题；证据不足时明确标注"需人工确认"
   - 行号只能来自任务文件中"diff 行号速查表"列出的 L<行号>，禁止估算

6. 末尾附一段**总评草稿**（可直接粘贴到 PR 评论区）。

7. 附"行级评论候选定位"小节，列出文件路径和对应新行号范围。

## 输出约束

- 默认中文简体输出
- 不输出空泛规范类建议，优先覆盖设计、安全、性能、稳定性问题
- 每条修改建议必须可落地，包含最小改动代码示例（行内代码格式）
- 若未发现实质性缺陷，明确写"未发现实质性缺陷"并补充剩余测试风险

## 异常处理

- 脚本失败：报告具体错误，提示检查 `.env` 中的 `GITCODE_TOKEN`、PR 链接格式
- 无输出文件：说明原因并重试一次
