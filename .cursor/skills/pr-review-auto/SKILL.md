# PR Review Auto

## 适用场景

- 用户给出 GitCode PR 链接（如 `.../pull/123` / `.../pulls/123`）
- 用户希望少步骤、一步到位拿到最终检视意见
- 需要输出固定模板：
  - `[文件:<path>] [行号:<start>-<end>] [严重级别:<高/中/低>] 【review】<一句话标题> <具体问题点 + 影响范围 + 触发场景，自然融入描述，禁止显式写「触发条件：」标签；证据不足时写"需人工确认"> 修改建议：<改法描述，代码必须使用行内代码 `改动示例` 格式，禁止只写文字>`

## 执行流程

1. 接收用户提供的 PR 链接。
2. 在项目根目录执行启动脚本：
   - **Windows**：`review.bat "<PR链接>" --backend agent`
   - **macOS / Linux**：`bash review.sh "<PR链接>" --backend agent`
   - 脚本会自动检测虚拟环境和可用的 Python 命令，无需手动判断。
3. 从 `output/` 目录读取最新生成的 `review_task_*.md` 与（若存在）`line_candidates_*.md`。
4. 基于任务文件与 diff，直接输出最终检视意见（无需用户再复制中间文件），要求：
   - 按严重级别排序（高/中/低）
   - 仅输出有证据支撑的问题；证据不足标注"需人工确认"
   - 每条必须严格按以下模板：

```text
[文件:<path>] [行号:<start>-<end>] [严重级别:<高/中/低>] 【review】<一句话标题> <具体问题点 + 影响范围 + 触发场景，自然融入描述，禁止显式写「触发条件：」标签；证据不足时写"需人工确认"> 修改建议：<改法描述，代码必须使用行内代码 `改动示例` 格式，禁止只写文字>
```

5. 在末尾附一段"总评草稿"（可直接粘贴到 PR）。
6. 增加"行级评论候选定位"小节（文件 + 新行号范围），便于用户回帖到具体行。

## 输出约束

- 默认中文简体
- 不输出空泛建议，不只复述规范
- 优先设计/安全/性能/稳定性问题
- 建议必须可执行，避免"建议优化一下"这类空话

## 快捷口令

- 触发短语：`/review <pr链接>` 或 `review <pr链接>`
- 示例：`/review https://gitcode.com/Ascend/msit/pull/5022`
- 收到以上任一口令时，按"执行流程"自动完成全链路并直接给出最终可复制意见。

## 异常处理

- 若命令执行失败：先报告错误原因，再给出下一步动作（如检查 `.env`、token、PR 链接格式）。
- 若未生成任务文件：明确提示并重新执行一次命令。
- 若脚本输出 `[SKIP]`：说明本次 PR diff 与上次完全相同，无需重新审查，直接告知用户。
